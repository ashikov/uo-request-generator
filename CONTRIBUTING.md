# Участие в разработке

Этот документ описывает практический порядок работы с репозиторием. Инженерные
правила находятся в [`AGENTS.md`](AGENTS.md), а правила оформления задач — в
[`docs/ISSUE_GUIDELINES.md`](docs/ISSUE_GUIDELINES.md).

## Перед началом работы

- Прочитайте [`README.md`](README.md) и актуальную документацию по затрагиваемой
  части проекта
- Ознакомьтесь с [`AGENTS.md`](AGENTS.md)
- Выберите существующую issue или создайте новую через шаблон GitHub
- Проверьте объём задачи, зависимости и пересечение с активными PR
- Не начинайте реализацию, если ожидаемый результат нельзя проверить по критериям
  готовности

## Работа с issue

Одна ветка и один PR обычно соответствуют одной issue.

Новая issue оформляется по
[`docs/ISSUE_GUIDELINES.md`](docs/ISSUE_GUIDELINES.md). Если во время работы
меняется объём задачи, после согласования нужно обновить описание issue, а не
оставлять решение только в комментариях или переписке.

## Ветка

Создавайте отдельную ветку от актуальной `main`.

Рекомендуемый формат имени:

```text
<type>/<issue-number>-<short-name>
```

Примеры:

- `feat/5-copy-result`
- `fix/12-provider-timeout`
- `docs/7-issue-guidelines`

Не смешивайте в одной ветке независимые изменения.

## Реализация

- Соблюдайте правила из [`AGENTS.md`](AGENTS.md)
- Разрабатывайте новую и изменяемую логику по TDD
- Не добавляйте абстракции и зависимости без необходимости
- Не выполняйте массовые косметические изменения вне объёма issue
- Поддерживайте код, тесты и документацию в согласованном состоянии
- Не добавляйте секреты, персональные данные и реальные пользовательские тексты

## Проверки

Перед открытием или обновлением PR выполните:

```bash
make lint-md
pnpm check
```

Если часть проверки невозможно выполнить локально, укажите это в описании PR и
объясните причину.

## Коммиты

- Используйте Conventional Commits
- Делайте коммиты атомарными
- Формулируйте сообщение по фактическому результату изменения
- Не включайте в коммит несвязанные файлы

Примеры:

```text
feat: add result copy action
fix: handle provider timeout
docs: document issue workflow
```

## Pull request

В описании PR укажите:

- что изменено
- как результат проверен
- какие ограничения или компромиссы остались
- изображения мобильного и настольного интерфейса для визуальных изменений
- `Closes #N` или `Fixes #N`, если PR полностью завершает issue

PR должен быть достаточно небольшим для последовательного ревью. Не добавляйте в
него изменения, которые не нужны для критериев готовности связанной issue.

Если реализация сознательно отклоняется от issue, сначала согласуйте решение и
обновите описание задачи.

## Приватность

В репозиторий, issue, PR, комментарии, тесты и примеры нельзя добавлять:

- личную переписку
- реальные адреса, фамилии, телефоны и номера квартир
- API-ключи и другие секреты
- необезличенные пользовательские тексты

Для примеров и фикстур используйте только вымышленные или полностью обезличенные
данные.
