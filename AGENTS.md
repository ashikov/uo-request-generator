# Правила работы с репозиторием

## Общие правила

- Использовать Conventional Commits
- Делать коммиты атомарными
- Не смешивать независимые изменения
- Перед завершением работы запускать `make lint-md` и `pnpm check`
- Не добавлять секреты
- Не менять продуктовые или архитектурные решения молча
- Существенные решения фиксировать в документации или ADR
- Не выдумывать недостающие факты
- Поддерживать код, тесты и документацию в согласованном состоянии
- Писать код и идентификаторы на английском языке
- Писать документацию и интерфейс на русском языке

## Архитектура и код

- Сохранять архитектуру простой
- Не создавать абстракции и пакеты «на будущее»
- Предпочитать встроенные возможности стека
- Веб-сервер реализовывать на Fastify
- Интерфейс реализовывать обычными HTML, CSS и минимальным browser JavaScript
- Не добавлять frontend-фреймворк без подтверждённой сложности интерфейса
- Не добавлять шаблонизатор, пока страница остаётся статической
- Не собирать HTML строками в TypeScript
- Не помещать доменную логику в Fastify-маршруты
- Не допускать зависимости `packages/core` от Fastify, HTTP или браузерного кода
- Предпочитать прямой и читаемый поток запроса скрытой магии фреймворка
- Избегать лишнего клиентского JavaScript
- Не добавлять runtime-зависимости без необходимости
- Держать конфигурацию явной
- Не использовать `any`
- Использовать TypeScript strict
- Разрабатывать новую и изменяемую логику по циклу TDD: падающий тест,
  минимальная реализация, рефакторинг на зелёных тестах
- Не логировать пользовательские тексты
- Не использовать LLM как источник фактов или законодательства

## Документация

- Использовать естественный и практичный язык
- Избегать корпоративного маркетингового языка и хайпа
- Один абзац должен выражать одну законченную мысль
- Не дублировать одну информацию по нескольким документам
- Не описывать будущие возможности как уже реализованные
- Не использовать реальные адреса, фамилии, телефоны и номера квартир в примерах
- Для конфигурационных файлов предпочитать YAML, если инструмент не требует
  другого формата или исполняемой конфигурации

## Списки

- Не использовать точки с запятой
- Короткие пункты оформлять без знака в конце
- Законченные предложения оформлять с точкой.
- Если пункт состоит из нескольких предложений, каждое предложение заканчивать
  точкой.
- Внутри одного списка сохранять единообразное оформление

## Окончание файлов

Markdown-файлы должны заканчиваться одной пустой строкой, то есть двумя
символами перевода строки после последней содержательной строки.

Остальные текстовые и кодовые файлы должны заканчиваться одним переводом строки
без дополнительной пустой строки.

Если в будущем появится `README.en.md`, его нужно поддерживать синхронно с
`README.md`. В текущем задании `README.en.md` не создавать.
