# Правила оформления issue

Этот документ определяет единый формат задач в репозитории. Он является источником
истины для структуры issue, размеров, зависимостей, критериев готовности и меток.
Инженерные правила разработки остаются в [`AGENTS.md`](../AGENTS.md).

## Язык

Issue оформляются на русском языке.

Названия файлов, команды, API-маршруты, переменные и другие технические
идентификаторы указываются в обратных кавычках: `pnpm check`,
`POST /api/generate`, `LlmGateway`.

## Заголовок

Формат заголовка:

```text
[Размер] Результат задачи
```

После размера используется глагол в неопределённой форме:

- `[S] Добавить кнопку копирования заявки`
- `[M] Подготовить MVP к проверке`
- `[L] Подключить OpenAI-совместимый LLM-коннектор`

Заголовок должен описывать ожидаемый результат, а не процесс работы.

Нежелательные варианты:

- `Поработать над UI`
- `Разные исправления`
- `Frontend`
- `Рефакторинг`

## Размер задачи

Размер показывает объём и связанность изменений, но не приоритет.

### `[S]`

Один локальный и однозначный результат. Изменения затрагивают небольшую часть
одного слоя и могут быть проверены независимо.

### `[M]`

Несколько связанных изменений внутри одного сценария или подсистемы. Может
потребоваться согласование нескольких файлов, тестов и документации.

### `[L]`

Сквозное или архитектурно значимое изменение с несколькими группами работ и
повышенным риском.

Задачу размера `[L]` следует проверить на возможность декомпозиции. Она остаётся
одной задачей только тогда, когда разделение создаст искусственные зависимости
или не позволит получить законченный результат.

## Одна задача — один результат

Issue должна приводить к одному законченному и проверяемому результату.

Связанные мелкие изменения можно объединять, если они относятся к одному
пользовательскому или техническому сценарию.

Не следует объединять в одну задачу работы, которые:

- имеют разные цели
- могут выпускаться независимо
- требуют разных исполнителей и почти не пересекаются по коду
- имеют разный порядок выполнения
- не могут быть проверены общими критериями готовности

## Обязательные разделы

### `## Цель`

Коротко объясняет, какой результат должен появиться и зачем он нужен. Цель не
должна быть списком файлов или технических действий.

### `## Входит в задачу`

Содержит конкретный объём работы, ограничения и обязательные проверки. Пункты
должны описывать существенные изменения, а не каждую будущую строку кода.

### `## Критерии готовности`

Содержит наблюдаемые условия, по которым можно однозначно понять, что задача
выполнена. Критерии не должны ограничиваться формулировками «код написан», «всё
работает» или «UI улучшен».

### `## Зависимости и параллельность`

Указывает:

- от каких issue или PR зависит задача
- что она блокирует
- какую часть можно выполнять параллельно
- где ожидается пересечение файлов или конфликт веток
- в каком порядке выполнять зависимые части

Если зависимостей нет, это указывается прямо.

## Дополнительные разделы

### `## Контекст`

Добавляется, если без него непонятна причина задачи, предыдущее решение или
важное ограничение. Контекст не должен превращаться в длинную историю проекта.

### `## Не входит в задачу`

Добавляется, если существует риск расширения объёма или смешения с соседними
issue. Особенно полезен для архитектурных задач, UI, рефакторинга и интеграций с
внешними сервисами.

### `## Порядок выполнения`

Добавляется, если внутри issue есть этапы, которые нельзя выполнять в
произвольной последовательности.

### `## Проверка`

Можно использовать для нестандартных ручных проверок, команд или сценариев,
которые неочевидны из критериев готовности.

## Зависимости

Зависимости указываются ссылками на конкретные issue или PR:

- `Зависит от issue #5`
- `Финальная проверка выполняется после PR #3`
- `Блокирует браузерную часть issue #6`

Не следует объявлять всю задачу заблокированной, если параллельно можно выполнить
подготовительную часть. Нужно отдельно описывать, что доступно сейчас, а что
требует завершения зависимости.

При ожидаемом пересечении необходимо перечислить основные конфликтующие файлы
или подсистемы.

## Критерии готовности

Каждый критерий должен быть проверяемым через одно или несколько действий:

- автоматический тест
- команда репозитория
- ручной сценарий
- проверка содержимого файла
- просмотр интерфейса
- отсутствие запрещённых данных или зависимостей

Критерии описывают результат, а не выбранный исполнителем способ реализации.

## Технические ограничения

В issue фиксируются только значимые ограничения:

- архитектурные границы
- требования безопасности и приватности
- запрет на новые зависимости
- совместимость API
- обязательные команды проверки
- требования к обработке ошибок
- требования к доступности

Не следует заранее диктовать имена внутренних функций, классов или мелкие детали
реализации, если они не являются частью контракта.

## Тесты и проверки

Для любых изменений кода issue должна требовать тесты изменённой логики либо
явно объяснять, почему автоматический тест неприменим.

Для проекта обязательной финальной командой остаётся:

```bash
pnpm check
```

Ручная проверка не заменяет автоматическую там, где поведение можно надёжно
проверить тестом.

## Документация

Обновление документации включается только тогда, когда задача изменяет:

- публичное или внутреннее поведение системы
- архитектуру
- настройку приложения
- способ запуска
- контракт API
- правила разработки

Не следует добавлять формальное обновление документации в каждую мелкую задачу.

## Приватность

Issue, комментарии, тесты и примеры не должны содержать:

- личную переписку
- реальные адреса
- телефоны
- фамилии
- номера квартир
- API-ключи
- пользовательские тексты, не прошедшие обезличивание

Для примеров используются вымышленные или обезличенные данные.

## Метки

На текущем размере проекта достаточно существующего набора меток.

Тип задачи:

- `enhancement` — новая возможность или улучшение
- `bug` — подтверждённая ошибка
- `documentation` — преимущественно изменение документации

Участие:

- `good first issue` — локальная задача без скрытых архитектурных зависимостей,
  подходящая новому участнику
- `help wanted` — задача, для которой действительно требуется или
  приветствуется отдельный исполнитель

Размер указывается в заголовке, поэтому отдельные метки `size:S`, `size:M` и
`size:L` пока не нужны.

Метки областей вроде `frontend`, `backend` или `llm` стоит вводить только после
появления нескольких одновременно открытых задач в каждой области.

## Приоритет

Размер задачи не означает её приоритет.

Пока backlog небольшой, порядок работы указывается через зависимости и текст
issue. Отдельные метки приоритета следует вводить только тогда, когда открытых
задач станет достаточно много и их порядок перестанет быть очевидным.

## Связь с pull request

PR должен ссылаться на issue через `Closes #N` или `Fixes #N`, если он полностью
завершает задачу.

Одна issue обычно закрывается одним PR. Несколько PR допустимы для большой задачи,
если в issue заранее указан порядок частей и понятно, какой PR закрывает её
окончательно.

Решения, меняющие объём задачи, фиксируются не только в комментариях: после
согласования описание issue обновляется и остаётся актуальным источником
требований.
